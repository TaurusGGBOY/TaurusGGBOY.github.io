2026 年 3 月 31 日，Anthropic 发布了 Claude Code 2.1.88。不到两个小时，这个版本里的源码就被人公开了。随后发生的事情包括撤包、GitHub 大规模下架，以及攻击者借“泄露版 Claude Code”传播窃密木马。

那么，一个用于调试的 `.map` 文件，为什么会产生这么长的一条影响链？

我们先看发布物本身。实际上，这次事件不需要假设 Anthropic 的内部网络被攻破，也没有证据支持这种说法。问题出在正常的 npm 发布流程：本来应该留在构建环境里的源码文本，被 source map 带进了公开软件包。

## 2026 年 3 月 31 日 06:36：问题从一次 npm publish 开始

[npm registry](https://registry.npmjs.org/@anthropic-ai%2fclaude-code) 记录的发布时间是 2026 年 3 月 30 日 22:36:48 UTC，换成北京时间就是 3 月 31 日 06:36。`@anthropic-ai/claude-code@2.1.88` 在这个时间进入公共仓库。

这个包里同时出现了 `cli.js` 和 `cli.js.map`。前者是打包后的程序，后者负责把打包代码映射回源文件。source map 本身并不等于源码泄露，关键要看它有没有携带 `sourcesContent`。如果这个字段只有文件路径，外部人员仍然拿不到原文；如果里面保存了完整源码，情况就完全不同了。

我检查了原始发布物中的 `cli.js.map`。文件大小是 59,766,257 字节，里面有 4756 个 `sources` 条目，也有 4756 份非空的 `sourcesContent`。也就是说，这个 map 不只告诉调试器“源码文件叫什么”，还把对应文本一起交了出去。拿到 npm 包以后，不需要反编译 `cli.js`，按数组位置把路径和内容配对，就能恢复出可阅读的源码树。

一些外部报道把泄露规模写成约 51.2 万行代码、1900 个 TypeScript 文件，本仓库恢复出的 map 条目则是 4756 个。为什么数字不一样？因为统计口径不同。1900 更接近业务源码文件数，4756 还包含第三方依赖和生成内容。这里真正需要记住的不是哪一个数字更大，而是每一项 `sources` 都有对应的非空 `sourcesContent`。这个条件成立，源码恢复就已经具备了。

## 2026 年 3 月 31 日 08:23：从发布错误变成公开事件

包发布不到两个小时，安全研究者 Chaofan Shou 在 [X 上公开了这个问题](https://x.com/Fried_rice/status/2038894956459290963)。帖子指出 Claude Code 的源码通过 npm 中的 map 文件暴露，还给出了 Anthropic Cloudflare R2 上的源码压缩包地址。这个帖子到现在显示约 3573 万次浏览。

我们可以把泄露过程简化成一条链路：Anthropic 发布 npm 包，包内 map 暴露源码文本，研究者公开可直接下载的地址，其他人开始保存和镜像。前两步属于发布流程问题，后两步则让它进入不可逆的传播阶段。

那么，Anthropic 关闭 R2 链接或者删除 npm 版本有没有用？有用，但只能阻止新的用户继续从原地址下载。已经保存到本地、同步到 GitHub 或重新打包的副本，不会因为上游链接失效而自动消失。这也是为什么后面的处理重点很快从“修包”转向“清理副本”。

## 2026 年 3 月 31 日当天：撤回版本，再处理 GitHub 镜像

Anthropic 随后撤下了 2.1.88。今天查看 npm registry，仍然可以看到这个版本的发布时间，但相邻版本都有的 tarball 地址、完整性签名和文件统计已经不存在。一个公开的 [Claude Code issue](https://github.com/anthropics/claude-code/issues/41666) 也把 2.1.88 称为 yanked release，并记录了 Skills、MCP 和自定义命令失效等用户反馈。

这里要区分两个问题。源码泄露解释了版本为什么必须立即撤回，但 Skills 和 MCP 失效属于版本质量问题。它们出现在同一个版本、同一天影响用户，却不是同一个故障机制。把它们放在一起讨论，是因为用户感受到的是同一条发布链路失灵：自动更新拿到了一个功能异常的版本，随后这个版本又因为源码暴露被撤下。

撤包之后，Anthropic 同一天向 GitHub 提交了 [DMCA 下架通知](https://github.com/github/dmca/blob/master/2026/03/2026-03-31-anthropic.md)。通知确认这批代码的版权属于 Anthropic，并要求删除主要镜像及其 fork。GitHub 在文档顶部说明，目标仓库网络超过 100 个，而且 Anthropic 认为大部分 fork 与源仓库存在相同侵权，因此平台最初对包含 8100 个仓库的整个网络执行了处理。

这个动作说明了两个事实。第一，这批代码虽然已经公开可下载，但没有因此变成开源软件。第二，npm 撤包只能处理发布源，无法处理已经扩散到其他平台的副本，所以 Anthropic 还需要借助版权程序继续清理。

## 2026 年 4 月 1 日至 2 日：技术事故开始影响公司形象

[《华尔街日报》4 月 1 日的报道](https://www.wsj.com/tech/ai/anthropic-races-to-contain-leak-of-code-behind-claude-ai-agent-4bc5acc7)称，Anthropic 最初的请求让 GitHub 移除了 8000 多份副本和改编版本，随后又把范围缩小到 96 个，因为第一次处理影响了超出预期的 GitHub 账户。

也就是说，DMCA 虽然快速压缩了主要镜像网络，却产生了新的问题：清理范围过大。Anthropic 不仅要解释为什么正式发布物会带上源码，还要解释为什么版权请求波及了这么多账户。影响到这里已经超出工程团队内部的打包错误，进入平台治理和开发者关系层面。

4 月 2 日，[Business Insider](https://www.businessinsider.com/anthropic-copyright-infringement-claude-code-2026-4) 引述 Anthropic 发言人称，公司正在推出措施防止类似事件再次发生。报道还指出一个比较尴尬的背景：Anthropic 长期面对训练数据版权诉讼，这次却需要站在版权拥有者的位置，用 DMCA 阻止别人继续传播它的代码。于是，一个 source map 配置问题，又被放进了 AI 公司如何对待他人作品和自身知识产权的讨论里。

不过，影响范围仍然需要说清楚。这次暴露的是 Claude Code 的客户端 Agent harness，包括工具调用、权限、状态和宿主逻辑，不是 Claude 模型权重。[Zscaler 的安全分析](https://www.zscaler.com/blogs/security-research/anthropic-claude-code-leak)也明确指出，没有证据显示模型权重、安全流水线或用户数据随之暴露。因此，这件事会让竞争者和安全研究者更容易理解 Claude Code，却不能推出“Claude 模型已经被完整公开”。

## 2026 年 4 月 3 日至 7 日：真正的下游风险开始出现

源码已经撤回，GitHub 镜像也在清理，那么风险是不是到这里就结束了？实际上没有。

[Trend Micro 4 月 3 日的调查](https://www.trendmicro.com/en_gb/research/26/d/weaponizing-trust-claude-code-lures-and-github-release-payloads.html)发现，攻击者在公开披露后的 24 小时内建立了假冒的“Claude Code 泄露版”仓库。这些仓库声称提供无限制版本或解锁企业功能，下载内容实际包含 Vidar 信息窃取程序和 GhostSocks 代理木马。

这里的攻击机制并不复杂。用户看到源码泄露新闻后，会主动搜索 GitHub 镜像；攻击者只需要准备一个名字相似的仓库、一份看起来像源码的 README，再把恶意可执行文件放进 Release 压缩包，就能利用这波搜索流量。Trend Micro 的判断是，这批攻击者原本就在轮换使用二十多个热门软件品牌，Claude Code 只是当时最容易吸引开发者的新诱饵。

到 4 月 7 日，[Trend Micro 的后续监测](https://www.trendmicro.com/en_gb/research/26/d/claude-code-remains-a-lure-what-defenders-should-do.html)仍把它标记为活跃攻击。被跟踪的仓库当时已经有 838 个 star、1060 个 fork，新的恶意压缩包确认下载 533 次，之前被替换或删除的链接还无法统计。后续样本又加入了 PureLog Stealer。

因此，泄露给 Anthropic 带来的安全影响分成了两层。源码本身让外部人员更容易研究 Agent 的攻击面，泄露事件的热度又给恶意软件提供了可信的传播外壳。前一层是代码和架构风险，后一层是品牌与供应链风险。修复 npm 包只能解决前者的一部分，解决不了已经形成的搜索关键词和社会工程场景。

## 2026 年 4 月 23 日：官方复盘了质量问题，没有复盘源码泄露

4 月 23 日，Anthropic 发布了 [Claude Code 质量问题复盘](https://www.anthropic.com/engineering/april-23-postmortem)。文章承认 3 月到 4 月间有三项独立改动造成体验下降。其中，3 月 26 日上线的一项缓存优化会在会话空闲后反复清除旧 thinking，导致 Claude 遗忘上下文、重复执行，并造成更多缓存 miss。Anthropic 在 4 月 20 日前修复了这些问题，随后为所有订阅用户重置使用限额，还承诺让更多内部员工使用与用户完全相同的公共构建版本，并加强 Code Review、评测和渐进发布。

这篇文章值得放进时间线，但不能把它当作源码泄露复盘。官方没有说缓存问题导致了 `cli.js.map` 被发布，也没有说后面的测试措施专门针对泄露事故。两件事的共同点在于，它们都发生在同一段发布周期，也都暴露了内部测试与公共发布之间的差距。

截至今天，我没有找到 Anthropic 针对源码泄露发布的独立技术复盘。公开可确认的回应主要来自 npm 撤包、GitHub DMCA，以及媒体转述的“正在采取措施防止再次发生”。至于究竟是哪条构建规则、文件过滤配置或发布检查失效，Anthropic 还没有公开到这个粒度。

## 2026 年 7 月 16 日：这次泄露到底留下了什么

npm registry 当前的 `latest` 是 2.1.211，`stable` 是 2.1.204，说明 Claude Code 的版本发布没有因为这次泄露停止。2.1.211 的 npm 包只有 7 个文件，解包大小约 160 KB；2.1.87 则有 19 个文件，解包后约 46.6 MB。发布形式已经发生了明显变化，但 Anthropic 没有公开说明这个变化由源码泄露直接推动，因此这里只能确认现象，不能确认因果。

回到开头的问题，一个 source map 为什么会产生持续几个月的影响？因为它打通了四个原本分开的环节。构建系统把源码写进 map，npm 把 map 公开分发，社交平台和 GitHub 让源码快速复制，攻击者又利用事件热度包装恶意下载。任何一个环节单独看都不复杂，连起来以后，撤回一个 npm 版本就不可能让事情恢复原状。

到今天，可以确认的直接影响是 Anthropic 撤回了正式版本，处理了大规模 GitHub 镜像，公开承诺防止重演，并承受了版权、发布质量和品牌安全方面的压力。没有公开证据表明 Claude 模型权重、用户数据或 Anthropic API 因此失守，也没有可靠数据可以计算收入损失或客户流失。

所以，研究这次事件时，最重要的边界是把三件事分开：公开发布事故可以确认，源码传播和恶意利用可以追踪，Anthropic 内部损失只能在有证据时讨论。

## 整个系列会讲什么

这一篇 00 先解决源码从哪里来的问题。接下来 47 篇不会按照目录顺序逐个介绍文件，而是沿着一次请求的执行过程向外展开。我们先建立系统模型，再进入 Agent 内核，然后补齐安全、上下文、扩展、宿主和产品能力。下面逐篇说明每篇文章要回答的问题。

### S0：先建立阅读边界和系统全景

**00 系列导读**就是你正在读的这一篇。它还原源码泄露的时间线，解释 source map 为什么能恢复源码，并划清静态源码能够证明什么。

**01 架构全景**会把 Claude Code 拆成入口、查询内核、执行与状态、扩展与远程运行几个部分。它先给后续文章建立坐标，避免看到一个函数名就猜整个系统的职责。

### S1：一条请求怎样进入 Agent 内核

**02 请求链路**从一次用户输入开始，跟踪上下文组装、模型流、`tool_use`、工具结果和最终输出，先把最重要的执行主线跑通。

**03 启动流程**把时间往前移动，研究第一条请求到来之前发生了什么，包括 CLI 入口、setup、配置预取、迁移、会话准备和 REPL 挂载。

**04 运行模式**比较交互式 CLI、print/headless、Agent SDK、MCP server、Bridge 和 direct-connect，解释不同入口为什么可以共享同一套内核。

**05 QueryEngine**研究会话的编排边界。它接收 prompt、工具、权限和状态，把宿主提供的能力整理好，再交给下层查询循环。

**06 Agent Query Loop**继续进入真正的循环，说明模型响应、工具调用、继续推理、预算和停止条件怎样共同决定一轮任务何时结束。

**07 消息模型**解释 user、assistant、tool 和 system 消息怎样关联，流式内容块又怎样变成内部可以继续处理的事件。

**08 API 流式调用**沿网络边界研究请求构造、流式与非流式响应、缓存、重试、错误映射和多后端适配。

**09 Tool 抽象**回答工具怎样被描述和发现。这里会分析 Tool 接口、输入 Schema、能力元数据、`buildTool` 和动态工具注册。

### S2：工具执行以后，系统怎样守住边界

**10 工具调度**研究一次响应里出现多个 `tool_use` 时，哪些调用可以并行，哪些必须串行，以及取消和进度怎样合并。

**11 工具生命周期**跟踪一次工具调用从查找、参数校验、授权、执行到结果持久化的完整过程，把调度层和具体副作用接起来。

**12 权限引擎**解释 allow、ask 和 deny 怎样匹配，多个权限来源冲突时按什么顺序决策，用户确认又怎样回到执行上下文。

**13 Sandbox 与 Bash**进入命令执行边界，研究 Bash 解析、路径校验、危险操作识别、沙箱和不同平台之间的差异。

**14 文件工具**分析 Read、Edit、Write 和 Notebook 怎样处理文件状态、并发修改、历史快照与回滚。

**15 搜索工具**比较 Glob、Grep、ripgrep、WebSearch 和 WebFetch，重点不是命令名称，而是搜索范围、分页、截断与失败恢复。

**16 系统提示词**研究 system prompt、CLAUDE.md、项目指令和环境上下文怎样分块，再在请求前组装成模型实际收到的上下文。

**17 上下文压缩**回答长会话装不下以后怎么办，包括 token 估算、microcompact、partial/full compact 和摘要回填。

**18 Hooks**解释 PreToolUse、PostToolUse、Stop 和 PreCompact 怎样横切运行时，在不同生命周期节点阻断、改写或补充信息。

**19 错误恢复**把正常路径放到一边，专门研究 API、流、工具和会话层的错误分类、退避、降级与恢复。

**20 会话历史**研究 JSONL transcript 怎样写入，`resume` 和 `fork` 怎样恢复或分叉对话，以及历史状态能恢复到什么程度。

### S3：能力怎样扩展到单个 Agent 之外

**21 Command 系统**比较 prompt、local 和 local-jsx 命令，解释命令的注册、解析、启用条件和 UI 执行流程。

**22 Skill 系统**研究 Skill 怎样被发现、加载和命令化，以及内联执行、fork 执行和插件注入之间有什么区别。

**23 Task 运行时**分析统一 Task 抽象怎样管理前台、后台、状态机、输出流、取消和终态。

**24 Subagent**研究任务怎样委派给隔离上下文，模型和工具如何选择，执行结果又怎样回到主 Agent。

**25 Agent Teams**把范围扩展到多个智能体，解释 teammate、消息传递、共享任务和协调器怎样组织协作。

**26 Plan Mode 与 Worktree**研究规划和执行怎样分开，退出计划为什么需要审批，worktree 又怎样隔离实际修改。

**27 MCP 集成**沿外部协议边界分析配置、连接、认证、工具与资源装配、权限和断线恢复。

**28 插件系统**研究插件的发现、安装、启停与 manifest，重点解释命令、Agent 和 Skill 注入以后怎样继续守住信任边界。

**29 LSP 集成**说明语言服务器怎样启动，诊断信息怎样进入注册表，并最终变成 Agent 可以使用的代码反馈。

**30 外部工具集成**研究浏览器、IDE 和其他宿主怎样与 Claude Code 通信，以及这些适配为什么不需要重新实现 Agent 内核。

### S4：同一个内核怎样服务不同宿主

**31 AppState**研究共享状态怎样创建、选择和订阅，以及 UI、工具和任务为什么可以读取同一份状态而不直接依赖彼此。

**32 Ink TUI**进入交互式终端，分析自定义 Ink/React 渲染、REPL、消息列表、输入框、弹窗和流式刷新。

**33 快捷键与 Vim**研究终端按键怎样解析、冲突怎样校验，以及 Vim 编辑模式如何用状态机处理不同上下文。

**34 无头 SDK**解释 print 模式、SDK 消息、JSON、SSE 和 WebSocket 怎样提供结构化输入输出，并处理没有人工确认的权限场景。

**35 设置与 Feature Flags**梳理用户、项目和托管配置层级，研究配置同步、缓存、实验开关与构建裁剪。

**36 模型路由**研究模型选择、认证和降级，以及 Anthropic、Bedrock、Vertex、Foundry、代理与 TLS 怎样接入同一请求层。

**37 Bridge 与 Remote**解释远程会话、WebSocket、worker 轮询、权限桥和 direct-connect server 怎样跨进程传递消息与状态。

**38 可观测性**研究日志、遥测、成本、usage、性能指标和诊断信息怎样记录，同时说明隐私开关会裁掉什么。

**39 更新与迁移**回到产品启动边界，分析自动更新、配置迁移、首次使用、项目信任和 onboarding 怎样保持兼容。

### S5：Agent 内核之上的产品与实验能力

**40 Session Memory**研究一次会话结束后，系统怎样提取并更新可以在以后复用的知识。

**41 Memdir**把记忆扩展到目录和团队范围，解释索引、相关内容检索、年龄策略、同步与密钥扫描。

**42 AutoDream**研究后台记忆整合怎样触发，锁、子 Agent、提示词和进度监视怎样避免它干扰主任务。

**43 Assistant 与 KAIROS**比较辅助模式、后台 Agent 和主 Coding Agent 的会话与消息路径，确认它们共享什么、又隔离什么。

**44 Buddy**分析陪伴式状态和交互怎样叠加在 Agent 之上，同时保持与核心执行逻辑分离。

**45 Voice**研究音频捕获、流式 STT、关键词、权限和状态怎样把语音输入接入终端 Agent。

**46 MagicDocs**解释文档生成、PromptSuggestion 和 tips 怎样主动提出内容，同时受当前项目上下文约束。

**47 通知与 Mailbox**收束通知、mailbox、队列消息和 output styles，研究这些非核心反馈通道怎样与任务状态协作。

整个系列仍然只讨论 `@anthropic-ai/claude-code@2.1.88`。源码能够直接确认的内容写成事实，调用关系连接出的职责写成解释，缺少运行证据的部分明确标成推断。顺序看起来很长，实际始终围绕同一个问题：一段用户输入，怎样经过模型、工具、权限、状态和宿主，最终变成一次可恢复、可扩展的 Agent 执行。
