把数据库 DDL 和用户问题一并交给大模型，一下午就能做出 Text2SQL Demo。真正困难的是第二天：同事问“上个月华东区新客的 7 日复购率”，模型生成的 SQL 能执行，也返回了一个像模像样的数字，但它把“新客”理解成首次下单用户，把“复购”理解成再次支付，把“华东区”理解成收货地址，而公司的指标口径完全不是这样。

Text2SQL 最危险的错误往往不是语法错误，而是**一条可以执行、结果却不正确的 SQL**。

这篇文章不从某个模型或框架开始，而是假设我们真有一个需求：为电商公司的分析师和运营人员做一个自然语言查数助手。然后像搭积木一样，每次只加入一种能力，观察它可能把执行效果推到什么位置，直到得到一套可以上线的实现。

文中的数值分为两类：论文或官方榜单成绩会明确给出来源；项目各阶段的区间则是工程目标，用于估算投入产出，不是把几篇论文的提升幅度机械相加。后者必须在自己的数据集上验证。

## 先定义我们的 Text2SQL 需求

假设公司数仓有 120 张表、约 1,600 个字段，覆盖订单、商品、用户、营销、售后和流量六个主题域。每天有约 3,000 次查询，其中 70% 是重复或轻微改写的经营问题，20% 是跨主题分析，10% 是第一次出现的长尾问题。

第一版只支持查询，不做图表和分析报告。系统输入一个自然语言问题，输出 SQL、查询结果和所采用的指标口径。

我们先整理 400 道离线题：

- 220 道常见经营问题，例如 GMV、退款率、客单价和渠道转化率；
- 100 道复杂问题，包含多表连接、子查询、窗口函数、同比或留存；
- 50 道带有实体和值映射的问题，例如“华东”“抖音渠道”“会员商品”；
- 30 道表述不完整的问题，例如“最近卖得最好的是哪个”，但没有说明时间、区域和“最好”的口径。

每道题不只保存一条标准 SQL，还保存预期结果、必须使用的指标定义、允许的等价写法和测试数据快照。因为两个 SQL 的文本不同，执行结果仍可能相同；反过来，在一个碰巧为空的数据集上，错误 SQL 也可能得到和标准 SQL 一样的结果。

主指标使用**结果正确率**：SQL 在固定快照上执行后，结果与预期答案一致，并通过口径断言。辅助指标包括：

- 可执行率：生成的 SQL 能否通过解析并成功运行；
- 复杂题正确率：只统计 100 道复杂问题；
- Schema Recall：生成正确 SQL 所需的表和字段是否都被检索到；
- P95 延迟：95% 请求在多少秒内返回；
- 单次相对成本：把最初版本记为 `1.0x`；
- 自动回答率：系统直接给出结果，而不是要求用户补充条件。

“可执行率”只能用来定位问题，不能当北极星指标。一个系统完全可能有 98% 的可执行率和 55% 的结果正确率。

## 阶段 0：把全量 DDL 交给一个强模型

最小实现非常简单：把 120 张表的建表语句、SQL 方言和用户问题拼成提示词，要求模型只输出 SQL。

```python
prompt = f"""
You are a SQL expert. Generate one read-only SQL query.
Dialect: Snowflake SQL

Schema:
{all_ddl}

Question:
{question}
"""

sql = llm.generate(prompt)
rows = warehouse.execute(sql)
```

这个版本适合建立基线，不适合继续堆提示词。全量 Schema 中存在大量同义字段，例如 `paid_at`、`payment_time`、`order_paid_time`；模型不仅要理解问题，还要在上千个字段中完成检索，注意力很容易浪费在无关内容上。

我们的工程预期是：整体结果正确率 **45%～55%**，复杂题 **25%～35%**，可执行率 **80%～90%**，P95 延迟约 **6～10 秒**，成本记为 `1.0x`。

这个起点并不悲观。BIRD 包含 95 个数据库、12,751 组问题与 SQL，早期 GPT-4 的执行准确率只有约 49%；截至 2026 年 7 月 20 日，BIRD 总榜领先系统的测试集成绩是 81.95%，人类参考成绩是 92.96%。两者之间的差距主要不是“换了一个更会写 SQL 的模型”，而是整个系统工程的差距。[BIRD 官方榜单](https://bird-bench.github.io/)

更接近企业环境的 Spider 2.0 包含 632 个真实工作流问题，数据库经常有上千个字段，并引入多种 SQL 方言和长查询。其早期基线中，o1-preview 也只有 17.1%。它提醒我们：传统公开集上的高分，不能直接等价为企业数仓里的成功率。[Spider 2.0](https://spider2-sql.github.io/)

## 阶段 1：先做语义层，再做 Schema 检索

第一个有效优化不是改 Prompt，而是整理元数据。

原始 DDL 只能说明字段类型和外键，不能说明 `net_gmv` 是否含税、退款应该按申请时间还是完成时间、用户归属区域取注册地还是常用收货地。我们为表和字段补充业务描述、同义词、单位、枚举值、关联关系和指标公式，再把整个数仓划分为六个主题域。

一次查询先经过主题路由，再检索候选表和字段：

```text
用户问题
  → 主题域路由
  → 表级召回
  → 字段级召回
  → 关系补全
  → 生成最小 Schema 上下文
```

这里要同时优化准确率和召回率。检索出 30 个无关字段通常只是增加噪声；漏掉一个必须参与连接的键，则注定无法生成正确 SQL。上线前可以把 Schema Recall 的门槛设为 **95%**，再逐步压缩候选集合。

CHESS 在大型数据库上采用分层检索和 Schema 选择，论文报告它把提示词 Token 数减少约 5 倍，同时准确率仍有提升；这比“把所有 DDL 塞进长上下文”更接近工业实现。[CHESS](https://arxiv.org/abs/2405.16755)

加入语义层和 Schema 检索后，我们把整体正确率目标设为 **58%～65%**，复杂题 **38%～48%**，上下文 Token 减少 **70%～85%**，P95 降到 **5～8 秒**，成本约 `0.6x～0.8x`。这一阶段最大的收益不是模型更聪明，而是模型面对的问题变小了。

如果 Schema Recall 还不到 90%，不要急着加入自我反思或多候选。生成器不可能使用一个它根本没看到的字段。

## 阶段 2：检索数据值，而不只是表名和字段名

用户不会按照数据库里的写法提问。

他说“抖音”，表里可能存的是 `channel_code = 'DY'`；他说“华东”，公司的区域维表可能把安徽划入华中；他说“会员商品”，真正条件可能是 `product_tag_id IN (...)`。仅凭字段描述，模型很容易写出语法正确但结果为空的 SQL。

因此，第二个检索器负责值链接：

1. 从问题中抽取实体、时间和数值条件；
2. 在维表值、枚举字典和业务术语库中混合检索；
3. 返回规范值、对应字段和少量真实样例；
4. 对高基数字段只返回候选，不把整列数据放进提示词。

例如系统不只告诉模型存在 `dim_channel.channel_code`，还会提供：

```text
用户表达：抖音、抖音直播、DY
规范渠道：channel_code = 'DOUYIN'
匹配来源：dim_channel.channel_name = '抖音电商'
```

Google Cloud 总结的生产经验同样把数据值采样、业务术语、Schema 检索和查询历史视为不同的信息层，而不是一次向量搜索解决全部问题。[Google Cloud 的 Text-to-SQL 技术总结](https://cloud.google.com/blog/products/databases/techniques-for-improving-text-to-sql)

完成值链接后，50 道实体映射题应从大约 40%～55% 提升到 **75%～85%**；整体正确率目标可以来到 **64%～71%**。P95 可能增加 0.5～1 秒，但由于 Schema 上下文更小，总成本仍可以控制在 `0.7x～0.9x`。

这一阶段要特别关注错误匹配。值检索 Top-1 不确定时，应把 2～3 个候选连同含义交给后续规划，而不是偷偷替用户做决定。

## 阶段 3：加入已验证样例，但不要建立 Prompt 垃圾场

公司 70% 的问题是重复或改写，这意味着历史上的正确 SQL 是最有价值的上下文。但不能把所有“点赞过的 SQL”直接当样例，因为旧查询可能使用废弃表、过期口径或硬编码时间。

我们只收录经过数据负责人验证的问法—SQL 对，并将样例绑定到指标版本和 Schema 版本。检索时同时考虑：

- 语义是否相近；
- 涉及的主题域、表和指标是否相同；
- SQL 结构是否相似，例如同比、Top N、漏斗或留存；
- 样例当前是否仍有效。

每次只给 3～5 个高质量样例。选择的重点不是自然语言最相似，而是能否展示当前问题需要的 SQL 结构和业务口径。

工业产品也在走这条路。Databricks Genie 把示例查询、SQL 表达式、业务说明和 Trusted Assets 放进独立的知识存储，并要求用真实问题集持续评测，而不是依赖一段越来越长的系统提示词。[Databricks Genie 最佳实践](https://docs.databricks.com/aws/en/genie-agents/best-practices)

这一阶段对高频问题最有效。我们的目标是让 220 道常见经营题达到 **80%～88%**，整体正确率达到 **70%～76%**，复杂题达到 **50%～60%**。检索本身增加的成本很低，整体仍约为 `0.8x～1.0x`。

如果样例上线后指标下降，先检查三个问题：样例是否过期，检索是否只看文本相似度，模型是否在复制样例中的表名而不是迁移其结构。

2026 年的 ReViSQL 进一步说明了数据质量的影响。作者人工审查 BIRD 训练集的一个子集时，发现其中 61.1% 的样本需要修正；在相同强化学习设置下，仅改善数据质量就让单次生成提高了 8.2～13.9 个百分点。论文还报告了 93.2% 的专家复核 Mini-Dev 成绩，但这个数字来自清洗后的小型开发集，不能与 BIRD 完整测试榜直接比较。[ReViSQL](https://arxiv.org/abs/2603.20004)

## 阶段 4：先规划，再生成；执行失败后做定向修复

复杂问题通常不是 SQL 语法难，而是需要先拆清楚计算过程。

以“上季度通过直播首购的新客，30 天内复购率同比变化多少”为例，至少需要确定：首购用户集合、直播渠道定义、观察窗口、复购事件、同比对齐方式和分母。让模型一步直接输出 SQL，相当于要求它同时完成需求分析、Schema Linking 和代码生成。

我们把生成分成三步：

1. 生成结构化计划，列出指标、过滤条件、中间集合、连接路径和时间窗口；
2. 检查计划引用的字段是否存在、时间粒度是否一致；
3. 根据已确认计划生成 SQL。

```json
{
  "metric": "30d_repurchase_rate",
  "cohort": "first_paid_order via livestream",
  "period": "previous_quarter",
  "numerator": "users with another paid order within 30 days",
  "denominator": "users in cohort",
  "compare": "same quarter last year",
  "join_path": ["fact_order", "dim_channel", "dim_user"]
}
```

生成的 SQL 先经过解析器和 `EXPLAIN`/Dry Run。数据库如果返回“字段不存在”“聚合层级错误”或“类型不匹配”，修复器会拿到错误位置、相关 Schema 和原始计划，只修改对应片段，最多重试两次。不要只给模型一句“SQL 错了，请重试”，那通常会得到一条变化很大、但不一定更正确的新 SQL。

DIN-SQL 的核心经验就是把 Schema Linking、复杂度判断、问题分解和自纠错拆开；论文报告这种分解方法相对未分解基线有约 10 个百分点的提升。[DIN-SQL](https://arxiv.org/abs/2304.11015)

最新研究也表明，这些结构化步骤不必依赖最大的模型。LitE-SQL 通过轻量 Schema 检索、两阶段训练和执行引导纠错，在 BIRD 开发集上达到 72.10%，参数量比对比系统少 2～30 倍。对企业项目而言，这意味着“上下文是否正确、流程是否可验证”常常比继续放大生成模型更值得优先投入。[LitE-SQL](https://aclanthology.org/2026.findings-eacl.186/)

我们的阶段目标是：可执行率达到 **95%～98%**，整体结果正确率 **75%～81%**，复杂题 **62%～72%**。P95 会升至 **8～12 秒**，成本约 `1.1x～1.4x`。

注意，可执行率的提升通常大于结果正确率。Dry Run 能发现不存在的列，却无法发现“复购率分母取错了”这种业务错误，所以后面仍需要结果级验证。

## 阶段 5：验证结果，不只验证 SQL

到这里，系统生成的 SQL 大多能够执行，但仍可能出现几类安静的错误：连接导致行数膨胀、`LEFT JOIN` 被过滤条件变成 `INNER JOIN`、同比少移了一年、金额单位放大 100 倍、分母为零，或者结果与常识严重冲突。

我们为每个候选 SQL 生成一份“结果证据”：

- 解析后的表、字段、过滤条件和连接图；
- Dry Run 扫描量和执行计划；
- 小样本执行结果；
- 行数、空值率、时间范围、最大最小值；
- 指标断言，例如比例应在 `[0, 1]`，订单金额不可为负；
- 与问题和规划逐项对齐的检查结果。

验证器不是再问一次同一个模型“你确定吗”，而是尽量提供新的、可验证的信息。若问题要求 2026 年 Q2，同比结果却只访问了 2026 年数据，系统可以确定它不正确；若连接后订单数变成主表的 12 倍，也应触发重新规划。

这一阶段预计不会大幅提高可执行率，但可以把整体正确率推到 **79%～84%**，复杂题达到 **68%～76%**。由于需要小样本执行和统计检查，P95 约 **9～14 秒**，仓库扫描成本也会增加。工程上应缓存中间结果，并限制验证查询的数据范围。

## 阶段 6：只对难题生成多个候选

单次生成的上限逐渐显现后，可以引入多路径推理，但不能让所有请求都生成 8 条 SQL。

系统先给问题估计难度。简单的单表聚合走单候选；涉及多主题、多时间窗口或窗口函数的题，分别从“按业务计划生成”“按相似样例改写”“按连接图生成”三条路径得到候选，再由验证器和选择器排序。

```text
简单问题 → 1 个候选 → 验证 → 执行

复杂问题 ─┬→ 业务计划路径 ─┐
          ├→ 样例改写路径 ─┼→ 执行验证 → 选择器 → 最终 SQL
          └→ 连接图路径   ─┘
```

CHASE-SQL 使用多路径推理和候选选择，在 BIRD 测试集上报告了 73.0% 的执行准确率；XiYan-SQL 也采用 Schema 筛选、多生成器、执行修复和候选选择，在 BIRD 上达到 75.63%。它们共同说明：候选之间真正有差异时，选择才有价值。[CHASE-SQL](https://arxiv.org/abs/2410.01943) · [XiYan-SQL](https://arxiv.org/abs/2507.04701)

更重要的工程原则是：不要用同一 Prompt、同一温度重复采样，再把多数票当作可靠性。三个高度相关的错误不会因为投票变成正确答案。

假设 25% 的请求被判为复杂题，并为它们生成 3 个候选，我们把整体正确率目标设为 **82%～87%**，复杂题 **74%～82%**。平均成本约 `1.4x～1.8x`，简单题 P95 不变，复杂题 P95 约 **15～25 秒**。

这是第一个明显的成本拐点。如果多候选只提升 1 个百分点，却让所有请求成本翻倍，应该回头改善语义层、检索和训练数据，而不是继续扩大采样。

## 阶段 7：把高频核心问题升级为可信 SQL

到这一步，我们需要接受一个事实：企业 Text2SQL 的最佳实践，不是让模型自由生成所有 SQL。

对于 GMV、利润、库存周转、广告 ROI 等高频核心指标，最可靠的做法是维护参数化 SQL 模板或经过验证的语义查询。模型只负责识别意图和填充时间、区域、渠道等参数，不重新发明指标公式。

```sql
-- trusted_query: gmv_by_region
SELECT
  region_name,
  SUM(net_paid_amount) AS gmv
FROM mart_order_metric
WHERE paid_date BETWEEN :start_date AND :end_date
  AND (:region IS NULL OR region_name = :region)
GROUP BY region_name
ORDER BY gmv DESC;
```

这不是倒退回固定报表。系统仍可以用自然语言匹配可信查询、补齐参数，并在可信查询无法覆盖时进入生成链路。二者是“确定性快车道”和“开放式慢车道”的关系。

Snowflake Cortex Analyst 使用语义模型描述业务概念，并通过 Verified Query Repository 保存经过验证的问题和 SQL；Databricks Genie 的 Trusted Assets 也提供参数化查询和受信任 SQL 函数。两家产品的方向高度一致：[Snowflake Cortex Analyst](https://docs.snowflake.com/en/user-guide/snowflake-cortex/cortex-analyst) · [Databricks Trusted Assets](https://docs.databricks.com/aws/en/genie-agents/tune-quality)

假设可信 SQL 覆盖 40% 的线上流量，这部分问题的结果正确率可以要求 **97%～99%**；剩余 60% 仍由生成链路处理。综合后，整体正确率目标约 **88%～92%**，常见经营题 **94%～97%**，复杂长尾题保持 **75%～83%**。由于大量高频请求走短路径，平均成本可回落到 `1.0x～1.3x`，常见题 P95 也能回到 **3～6 秒**。

这也是“研究榜单 80% 左右，为什么内部系统却可以把目标定到 90%”的答案：不是开放式生成突然超过了人类，而是企业系统缩小了问题域，用语义层约束口径，用可信 SQL 覆盖高频问题，并允许少量不完整问题先补充条件。两种数字不能直接比较。

## 最终架构

完整链路可以保持为八个清晰的组件：

```text
                   ┌───────────────可信查询命中──────────────┐
用户问题 → 意图/难度路由 → 语义目录与样例检索 → 参数填充 → 结果验证
                  │
                  └→ Schema + 值检索
                         → 结构化规划
                         → SQL 生成
                         → 解析 / Dry Run / 定向修复
                         → 单候选或多候选选择
                         → 结果验证
                         → 最终执行
```

各阶段的预期变化汇总如下。再次强调，这是我们为这个 120 表电商数仓制定的目标区间，不是论文结论。

| 阶段 | 新增能力 | 整体结果正确率 | 复杂题正确率 | P95 延迟 | 相对成本 |
| --- | --- | ---: | ---: | ---: | ---: |
| 0 | 强模型 + 全量 DDL | 45%～55% | 25%～35% | 6～10 秒 | 1.0x |
| 1 | 语义层 + Schema 检索 | 58%～65% | 38%～48% | 5～8 秒 | 0.6x～0.8x |
| 2 | 值与实体链接 | 64%～71% | 43%～53% | 6～9 秒 | 0.7x～0.9x |
| 3 | 已验证样例 | 70%～76% | 50%～60% | 6～10 秒 | 0.8x～1.0x |
| 4 | 规划 + Dry Run + 修复 | 75%～81% | 62%～72% | 8～12 秒 | 1.1x～1.4x |
| 5 | 结果级验证 | 79%～84% | 68%～76% | 9～14 秒 | 1.2x～1.5x |
| 6 | 难题多候选 | 82%～87% | 74%～82% | 15～25 秒（难题） | 1.4x～1.8x |
| 7 | 40% 流量走可信 SQL | 88%～92% | 75%～83% | 3～6 秒（常见题） | 1.0x～1.3x |

## 应该先做哪一步

如果从今天开始实现，我会按下面的顺序投入：

第一周，不优化模型。先收集 200～400 个真实问题，固定数据库快照，建立结果级评测；同时跑出全量 DDL 基线。

第二到三周，整理最常用主题域的语义层，完成表、字段、关系和值检索。只要 Schema Recall 还没稳定超过 95%，就继续修检索。

第四周，接入已验证样例和结构化规划，并把解析错误、执行错误、结果断言失败分开记录。不要把它们统称为“模型答错”。

第五周，只对复杂题开启多候选，并核算每提高 1 个百分点需要多少 Token、多少数据库扫描量和多少延迟。

之后，根据线上问题频率，把排名最靠前、口径最稳定的问题逐步沉淀为可信 SQL。每新增或修改一个指标，都把相应真实问题加入回归集。

模型当然重要，但在一个真实 Text2SQL 系统里，模型更像发动机，而不是整辆车。语义层决定它是否理解公司的语言，检索决定它看见什么，规划决定它如何拆题，验证决定错误能否在交付前被发现，可信 SQL 则决定高频核心结果能不能稳定复现。

从 50% 到 90% 的路径，不是寻找一条神奇 Prompt，而是逐步减少系统可以犯错的空间。
