如果现在要做一个 Text2SQL，第一步应该做什么？

很多人的答案是选模型、写 Prompt，或者先准备一批 SQL 做微调。我的答案更简单：**先准备一套能够判定结果对错的题。**

原因并不复杂。Text2SQL 最难发现的错误通常藏在语义里。模型写出的 SQL 可以执行，也确实返回了一个数字，但它把“新客”理解成首次下单用户，把“复购”理解成再次支付，把“华东区”理解成收货地址。三个词都解释得通，最后的结果却偏离了公司的指标口径。

如果没有固定的问题、数据快照和结果断言，我们甚至不知道下一次优化到底有没有效果。换了模型以后，可执行率可能上升，结果正确率却可能下降；加入多候选以后，正确率可能增加 2 个百分点，成本却增加两倍。

所以这篇文章从一套具体需求开始。我们先做出一个很差、但能测量的版本，再沿着一条查询链路逐步减少它可以犯错的地方。

文中有两类数字。论文与官方榜单成绩会给出来源。各阶段的准确率、延迟和成本区间，是本文项目的工程目标，用来决定下一步是否值得做。两类数字不能直接相加。

## 先把需求说清楚

假设我们要给一家电商公司做自然语言查数。数仓有 120 张表、约 1,600 个字段，分成订单、商品、用户、营销、售后和流量六个主题域。每天大约有 3,000 次查询。

这些查询并不平均。大约 70% 是 GMV、退款率、客单价、渠道转化率等重复问题；20% 需要跨主题分析；剩下 10% 是第一次出现的长尾问题。

系统收到一句自然语言，最终返回 SQL、查询结果和使用的指标口径。第一版只做这件事，不负责画图，也不继续解释经营原因。

我们从历史查询中整理 400 道题：220 道常见经营题，100 道多表连接、窗口函数、同比或留存题，50 道实体和值映射题，再加 30 道缺少时间或指标口径的问题。

每道题保存四样东西：标准结果、指标定义、允许的等价 SQL、固定的数据快照。这里只保存一条标准 SQL 还不够。两条文本完全不同的 SQL 可能得到同一个正确结果；一条错误 SQL 也可能因为当前数据为空，碰巧与标准 SQL 返回相同内容。

因此，主指标使用结果正确率：SQL 在固定快照上执行以后，结果与预期一致，同时通过指标断言。我们另外记录可执行率、复杂题正确率、Schema Recall、P95 延迟、Token 成本和数据库扫描量。

先把这几个指标分开很重要。一个系统完全可能有 98% 的可执行率，同时只有 55% 的结果正确率。

## 第 0 步：把所有 DDL 交给模型

我们先看最小版本。它把 120 张表的 DDL、SQL 方言和用户问题拼成一个 Prompt，然后直接执行模型生成的 SQL。

```python
prompt = f"""
You are a SQL expert. Generate one SQL query.
Dialect: Snowflake SQL

Schema:
{all_ddl}

Question:
{question}
"""

sql = llm.generate(prompt)
rows = warehouse.execute(sql)
```

这个版本的意义是建立基线。它的缺点也很直接：模型既要理解用户问题，又要在 1,600 个字段中完成检索，还要处理 `paid_at`、`payment_time`、`order_paid_time` 这类含义相近的字段。

我们把整体结果正确率目标设为 **45%～55%**，复杂题 **25%～35%**，可执行率 **80%～90%**。P95 大约 **6～10 秒**，成本记为 `1.0x`。

这个起点并不低。BIRD 包含 95 个数据库、12,751 组问题与 SQL，早期 GPT-4 的执行准确率约为 49%。截至 2026 年 7 月 20 日，BIRD 总榜领先系统的测试集成绩是 81.95%，人类参考成绩是 92.96%。[BIRD 官方榜单](https://bird-bench.github.io/)

Spider 2.0 更接近企业环境。它包含 632 个真实工作流问题，数据库经常有上千个字段，还引入多种 SQL 方言和长查询。它的早期基线中，o1-preview 只有 17.1%。传统公开集上的高分，放进真实数仓以后可能完全是另一回事。[Spider 2.0](https://spider2-sql.github.io/)

接下来要做的，就是把模型面对的 120 张表逐步缩小成当前问题真正需要的上下文。

## 第 1 步：先整理语义，再检索 Schema

原始 DDL 只能告诉模型字段叫什么、类型是什么。它不会说明 `net_gmv` 是否含税，退款按申请时间还是完成时间，用户区域取注册地还是常用收货地。

因此，我们先为表和字段补充业务描述、同义词、单位、枚举值、指标公式和关联关系，再把整个数仓划分成六个主题域。

一次查询会先判断主题域，再做表级召回和字段级召回，最后补齐连接所需的键：

```text
用户问题
  → 主题域路由
  → 表级召回
  → 字段级召回
  → 关系补全
  → 最小 Schema 上下文
```

这里最容易优化错的是候选数量。多召回 20 个无关字段会增加噪声，漏掉一个连接键则会让正确 SQL 失去生成条件。我们先把 Schema Recall 做到 **95%**，再压缩候选集合。

CHESS 采用分层检索和 Schema 选择。论文报告它把提示词 Token 减少约 5 倍，同时准确率仍有提升。这说明大型 Schema 的首要问题在于上下文是否相关，单纯扩大窗口解决不了它。[CHESS](https://arxiv.org/abs/2405.16755)

完成语义层和 Schema 检索以后，整体正确率目标来到 **58%～65%**，复杂题 **38%～48%**。上下文 Token 减少 **70%～85%**，P95 降到 **5～8 秒**，成本约 `0.6x～0.8x`。

如果 Schema Recall 还没有稳定超过 90%，先不要做自我反思和多候选。生成器不会使用它根本没有看到的字段。

### 要不要先做一张宽表

工业项目里还有一条更短的路。

假设大部分问题都在订单、用户和商品之间反复连接。我们可以提前建设 `mart_order_user_product`，把稳定的 JOIN 放进数据建模层。它可以是一张物理宽表，也可以是 dbt 或语义层维护的逻辑视图。

这样做以后，模型不需要每次重新推断三张表怎样连接。一个多表问题被降成了单表过滤和聚合。

不过，宽表也有自己的边界。字段太多会重新制造 Schema 噪声；不同事实粒度放在一起会重复计数；实时性不同的主题也很难稳定合并。

所以我们只在 JOIN 路径稳定、查询频率高、事实粒度清楚时建设主题宽表。对本文项目，可以把“被宽表覆盖的复杂题提高 **5～10 个百分点**”作为 A/B 目标。这个数字是工程假设，没有通用的公开基准。

## 第 2 步：把用户说的值映射到数据库

Schema 找对以后，SQL 仍然可能返回空结果。

用户说“抖音”，数据库里可能存的是 `channel_code = 'DOUYIN'`；用户说“华东”，公司区域维表可能把安徽划进华中；用户说“会员商品”，真正条件可能是一组 `product_tag_id`。

我们需要第二个检索器。它先从问题中抽取实体、时间和数值条件，再到维表值、枚举字典和业务术语库中检索规范值。

```text
用户表达：抖音、抖音直播、DY
规范渠道：channel_code = 'DOUYIN'
匹配来源：dim_channel.channel_name = '抖音电商'
```

这个过程称为值链接。它与 Schema 检索解决的是两个问题：Schema 检索决定用哪个字段，值链接决定这个字段应该与什么值比较。

Google Cloud 的生产方案也把数据值采样、业务术语、Schema 检索和查询历史拆成不同的信息层，分别处理各自的问题。[Google Cloud 的 Text-to-SQL 技术总结](https://cloud.google.com/blog/products/databases/techniques-for-improving-text-to-sql)

加入值链接以后，50 道实体映射题可以把目标从约 40%～55% 提高到 **75%～85%**，整体正确率来到 **64%～71%**。P95 增加约 0.5～1 秒，成本仍可以控制在 `0.7x～0.9x`。

如果 Top-1 候选不够确定，系统应该保留 2～3 个候选交给后续规划。此时直接替用户决定，往往会得到一条执行成功、业务含义错误的 SQL。

## 第 3 步：检索已经验证过的 SQL

现在回到前面的流量分布。70% 的问题是重复问题或轻微改写，历史上的正确 SQL 能提供比通用 Prompt 更具体的结构和口径。

但这里不能直接使用所有查询日志。旧 SQL 可能引用废弃表，可能使用过期口径，也可能把时间写死。我们只收录经过数据负责人验证的问法—SQL 对，并把它们绑定到指标版本和 Schema 版本。

检索样例时，我们同时看自然语言、主题域、指标和 SQL 骨架。一个“最近一个月销量最高的商品”与一个“最近一个月退款最多的渠道”在文字上并不相似，但它们可能共享 Top N 的 SQL 结构。

每次放入 3～5 个样例即可。样例选择策略通常比样例数量更重要。Databricks Genie 也把示例查询、SQL 表达式、业务说明和 Trusted Assets 放进独立的知识存储，并使用真实问题集持续评测。[Databricks Genie 最佳实践](https://docs.databricks.com/aws/en/genie-agents/best-practices)

这一阶段主要改善高频问题。220 道常见经营题的目标可以提高到 **80%～88%**，整体正确率达到 **70%～76%**，复杂题达到 **50%～60%**。整体成本约为 `0.8x～1.0x`。

如果加入样例以后指标下降，我们先检查三个地方：样例是否已经过期，检索是否只看文本相似度，模型是否在复制样例表名而没有迁移 SQL 结构。

ReViSQL 给出了一个更直接的提醒。作者人工检查 BIRD 训练集的一个子集，发现其中 61.1% 的样本需要修正。在相同 RLVR 设置下，仅改善数据质量就让单次生成提高 8.2～13.9 个百分点。论文还报告了 93.2% 的专家复核 Mini-Dev 成绩，但这个数字来自清洗后的小型开发集，不能与 BIRD 完整测试榜直接比较。[ReViSQL](https://arxiv.org/abs/2603.20004)

到这里，我们已经有了可以运行的 MVP。接下来的优化开始处理复杂问题。

## 第 4 步：先生成计划，再生成 SQL

考虑这个问题：

> 上季度通过直播首购的新客，30 天内复购率同比变化多少？

模型至少要确定首购用户集合、直播渠道、观察窗口、复购事件、同比对齐方式和分母。一步输出 SQL，相当于把需求分析、Schema Linking、时间推理和代码生成放在同一次采样里。

我们先让模型输出一个结构化计划：

```json
{
  "metric": "30d_repurchase_rate",
  "cohort": "first_paid_order via livestream",
  "period": "previous_quarter",
  "numerator": "users with another paid order within 30 days",
  "denominator": "users in cohort",
  "compare": "same quarter last year",
  "join_path": ["fact_order", "dim_channel", "dim_user"]
}
```

系统先检查计划引用的字段是否存在、时间粒度是否一致，然后才生成 SQL。DIN-SQL 的主要思路也是把 Schema Linking、复杂度判断、问题分解和自纠错拆开。论文报告这种分解方法相对未分解基线有约 10 个百分点的提升。[DIN-SQL](https://arxiv.org/abs/2304.11015)

SQL 生成以后先经过解析器和 `EXPLAIN` 或 Dry Run。数据库如果返回字段不存在、聚合层级错误或类型不匹配，修复器拿到错误位置、相关 Schema 和原始计划，只修改对应部分，最多重试两轮。

这里只给模型一句“SQL 错了，请重试”通常没有帮助。模型可能重写整条 SQL，同时引入新的语义错误。定向修复的关键，是把数据库刚刚提供的新信息交回去。

这一阶段把可执行率目标提高到 **95%～98%**，整体结果正确率 **75%～81%**，复杂题 **62%～72%**。P95 上升到 **8～12 秒**，成本约 `1.1x～1.4x`。

LitE-SQL 说明这些步骤不一定依赖最大的模型。它通过轻量 Schema 检索、两阶段训练和执行引导纠错，在 BIRD 开发集达到 72.10%，参数量比对比系统少 2～30 倍。[LitE-SQL](https://aclanthology.org/2026.findings-eacl.186/)

不过，Dry Run 只能发现数据库能够观察到的问题。它可以发现列不存在，无法发现复购率分母选错。下一步要检查的，是 SQL 的结果。

## 第 5 步：语法通过以后，再验证结果

一条可以执行的 SQL 仍然可能有很多安静的错误。

`LEFT JOIN` 可能被 `WHERE` 条件变成 `INNER JOIN`，连接可能让订单行数膨胀 12 倍，同比查询可能没有访问上一年数据，金额单位也可能被放大 100 倍。

因此，每个候选 SQL 都要生成一份结果证据。它包括解析后的表、字段、过滤条件和连接图，也包括 Dry Run 扫描量、小样本结果、行数、空值率、时间范围和最大最小值。

指标层再提供一些确定性断言。例如比例必须在 `[0, 1]` 内，订单金额不能为负，同比查询必须同时覆盖两个时间段。若问题要求 2026 年 Q2 同比，SQL 却只访问了 2026 年数据，系统可以直接判定它不完整。

验证器的价值来自新的证据。把同一条 SQL 再交给同一个模型，问一句“你确定吗”，通常只会放大模型原来的判断。

完成结果验证以后，整体正确率目标提高到 **79%～84%**，复杂题达到 **68%～76%**。P95 约为 **9～14 秒**。小样本查询和统计检查会增加仓库扫描，因此需要限制数据范围并缓存中间结果。

## 第 6 步：只给难题更多计算量

单次生成逐渐接近上限以后，可以让复杂问题生成多个候选。

简单的单表聚合仍然只生成一条 SQL。涉及多主题、多时间窗口或窗口函数的问题，分别从业务计划、相似样例和连接图三条路径生成候选：

```text
简单问题 → 1 个候选 → 验证 → 执行

复杂问题 ─┬→ 业务计划路径 ─┐
          ├→ 样例改写路径 ─┼→ 执行验证 → 选择器 → 最终 SQL
          └→ 连接图路径   ─┘
```

这里的重点是候选差异。如果三次调用使用同一个 Prompt、同一个模型和相近温度，得到的往往是三个高度相关的错误。多数票不会自动把它们变成正确答案。

CHASE-SQL 使用多路径推理和候选选择，在 BIRD 测试集报告了 73.0% 的执行准确率。XiYan-SQL 使用 Schema 筛选、多生成器、执行修复和候选选择，在 BIRD 上达到 75.63%。[CHASE-SQL](https://arxiv.org/abs/2410.01943) · [XiYan-SQL](https://arxiv.org/abs/2507.04701)

假设 25% 的请求属于复杂题，并为它们生成三个候选，整体正确率目标可以来到 **82%～87%**，复杂题 **74%～82%**。平均成本约 `1.4x～1.8x`，复杂题 P95 大约 **15～25 秒**。

这是一个明显的成本拐点。如果多候选只提高 1 个百分点，却让所有请求成本翻倍，我们应该回到语义层、检索和样例数据继续找问题。

## 第 7 步：高频问题走可信 SQL

走到这里会发现一个很实际的问题：GMV、利润、库存周转和广告 ROI 每天都有人问，我们为什么还要让模型每天重新生成它们的指标公式？

对于这些高频且口径稳定的问题，我们维护参数化 SQL 或经过验证的语义查询。模型负责识别意图和填充时间、区域、渠道等参数。

```sql
-- trusted_query: gmv_by_region
SELECT
  region_name,
  SUM(net_paid_amount) AS gmv
FROM mart_order_metric
WHERE paid_date BETWEEN :start_date AND :end_date
  AND (:region IS NULL OR region_name = :region)
GROUP BY region_name
ORDER BY gmv DESC;
```

系统仍然保留开放生成链路。可信查询命中时走短路径，无法覆盖时再进入规划、生成和验证。

Snowflake Cortex Analyst 使用语义模型描述业务概念，并用 Verified Query Repository 保存已验证的问题和 SQL。Databricks Genie 的 Trusted Assets 也提供参数化查询和受信任 SQL 函数。[Snowflake Cortex Analyst](https://docs.snowflake.com/en/user-guide/snowflake-cortex/cortex-analyst) · [Databricks Trusted Assets](https://docs.databricks.com/aws/en/genie-agents/tune-quality)

假设可信 SQL 覆盖 40% 的线上流量，这部分问题可以要求 **97%～99%** 的结果正确率。剩下 60% 继续走开放生成链路。综合以后，整体正确率目标约为 **88%～92%**，常见经营题 **94%～97%**，复杂长尾题 **75%～83%**。

高频请求走短路径以后，平均成本可以回落到 `1.0x～1.3x`，常见题 P95 回到 **3～6 秒**。

这也解释了一个看似矛盾的数字：研究榜单还在 80% 左右，企业内部系统却可以把目标定到 90%。内部系统缩小了问题域，用语义层固定口径，再用可信 SQL 覆盖高频流量。这个 90% 不能拿来证明开放式 SQL 生成已经达到同样水平。

## 什么时候开始训练模型

前面七步主要在改系统。什么时候应该从 Prompt、检索和编排转向 SFT？

我的判断条件有四个：开放生成链路已经达到约 75%～80%；主要错误不再来自检索漏表；已经积累至少 1,000 条经过复核的问法—SQL—结果样本；团队确实需要更低成本、更稳定的 SQL 风格或私有化部署。

在此之前训练，模型很容易把仍在变化的表结构和指标口径固化进去。

### SFT 学习什么

一条训练样本不应该只有问题和 SQL。它还要带上 Schema 版本、最小 Schema、业务证据、结构化计划、结果断言和难度标签：

```json
{
  "question": "上季度直播新客的 30 日复购率",
  "schema_version": "2026-07-15",
  "linked_schema": ["mart_order_user", "dim_channel"],
  "business_evidence": ["新客按首次支付定义", "直播渠道码为 LIVE"],
  "plan": ["构造首购 cohort", "检查 30 日窗口", "计算用户级比例"],
  "sql": "WITH cohort AS (...) SELECT ...",
  "result_assertions": ["rate >= 0", "rate <= 1"],
  "difficulty": "hard"
}
```

这样，模型会学习怎样从正确上下文得到正确 SQL，同时避免只背诵表名。

训练集按时间切分。若同一张报表的十种改写同时出现在训练集和测试集，离线结果会被高估。每条 SQL 还要在多个数据快照上执行，防止错误 SQL 因为空结果碰巧通过。

对本文项目，1,000～3,000 条高质量领域样本完成 SFT 后，可以把开放生成链路的增量目标设为 **3～8 个百分点**，同时把单次调用成本降低 **30%～60%**。这是项目验收目标，论文没有承诺同样的收益。

### RLVR 学习最终结果

SFT 让模型模仿正确答案。RLVR 直接用执行结果做奖励：候选 SQL 在受控快照上执行，结果与标准结果一致记 1，否则记 0。

Arctic-Text2SQL-R1 使用简单执行奖励，配合干净数据和 SFT 初始化取得了很强的效果，其 7B 模型超过此前部分 70B 级系统。[Arctic-Text2SQL-R1](https://arxiv.org/abs/2505.20315)

二值奖励的问题是过于稀疏。模型写对 90% 和完全写错都得到 0，很难知道哪次修正更接近答案。SQL-ASTRA 使用列集合匹配和轨迹聚合，在相同模型上相对二值 GRPO 报告了约 5% 的 BIRD 增益。Progress-SQL 则把奖励放到多轮修正过程，鼓励模型更早得到正确 SQL。[SQL-ASTRA](https://arxiv.org/abs/2603.16161) · [Progress-SQL](https://arxiv.org/abs/2606.06825)

最终结果仍然应该是主要奖励。列集合、语法有效性和修正进度只能提供辅助信号。否则模型可能选择了正确字段，却使用错误的聚合和过滤条件。

本文项目可以把 SFT 之后的 RLVR 增量目标设为 **2～5 个百分点**。如果同时使用多候选和迭代修复，需要分别报告单次生成与测试时扩展成绩。

Agentar-Scale-SQL 组合了 RL 增强的内部推理、顺序修正和并行候选选择，在 BIRD 测试集达到 81.67%。这是整套编排的成绩，不能全部归因于 GRPO。[Agentar-Scale-SQL](https://arxiv.org/abs/2509.24403)

训练提高开放式长尾生成能力，可信 SQL 提高高频问题的确定性。两条路线可以同时使用，收益仍然需要分别测量。

## 最终系统长什么样

把前面的步骤连起来，一次查询有两条路径：

```text
                   ┌──────────────可信查询命中──────────────┐
用户问题 → 意图/难度路由 → 语义目录与样例检索 → 参数填充 → 结果验证
                  │
                  └→ Schema + 值检索
                         → 结构化规划
                         → SQL 生成
                         → 解析 / Dry Run / 定向修复
                         → 单候选或多候选选择
                         → 结果验证
                         → 最终执行
```

各阶段的目标汇总如下。它们描述的是本文这套 120 表电商数仓，不能当作通用基准：

| 阶段 | 新增能力 | 整体结果正确率 | 复杂题正确率 | P95 延迟 | 相对成本 |
| --- | --- | ---: | ---: | ---: | ---: |
| 0 | 强模型 + 全量 DDL | 45%～55% | 25%～35% | 6～10 秒 | 1.0x |
| 1 | 语义层 + Schema 检索 | 58%～65% | 38%～48% | 5～8 秒 | 0.6x～0.8x |
| 2 | 值与实体链接 | 64%～71% | 43%～53% | 6～9 秒 | 0.7x～0.9x |
| 3 | 已验证样例 | 70%～76% | 50%～60% | 6～10 秒 | 0.8x～1.0x |
| 4 | 规划 + Dry Run + 修复 | 75%～81% | 62%～72% | 8～12 秒 | 1.1x～1.4x |
| 5 | 结果级验证 | 79%～84% | 68%～76% | 9～14 秒 | 1.2x～1.5x |
| 6 | 难题多候选 | 82%～87% | 74%～82% | 15～25 秒（难题） | 1.4x～1.8x |
| 7 | 40% 流量走可信 SQL | 88%～92% | 75%～83% | 3～6 秒（常见题） | 1.0x～1.3x |

如果只做小范围试用，做到第 3 步已经可以交付，目标约为 70%～76%。日常经营查数至少做到第 5 步，目标约为 79%～84%。高频核心指标和复杂长尾同时存在时，再加入多候选与可信 SQL。

SFT 和 RLVR 不属于固定阶段。它们是一条模型部署分支。即使训练没有达到目标，检索、验证和可信 SQL 也应该能够独立工作。

## 上线以后，错误要能够回到系统里

离线评测只能说明发布当天的能力。Schema 会变化，指标会改口径，用户也会问出测试集之外的问题。

所以每次请求要保存一条可复现的执行轨迹：

```text
request_id / question / timestamp
schema_version / metric_version
retrieved_tables / retrieved_columns / retrieved_values / retrieved_examples
structured_plan / generated_candidates / selected_sql
parser_result / dry_run_result / execution_result_hash
latency_by_stage / input_tokens / output_tokens / warehouse_scan_bytes
user_feedback / analyst_corrected_sql / error_category
```

`error_category` 至少区分漏表漏字段、值映射、JOIN、时间窗口、聚合与分母、SQL 方言、候选选择和指标口径过期。只记录一个“答案错误”，下一轮仍然不知道该改哪个组件。

每周从低置信度、用户点踩和结果断言失败的请求中抽样。分析师修正 SQL 与业务证据。修正样本先进入回归集；稳定且高频的进入示例库或可信 SQL；经过复核、去重的数据才进入 SFT 或 RLVR。

评测也要按问题切片。整体正确率从 84% 升到 85%，可能只是简单题变好。如果多表 JOIN 仍然停在 60%，用户体感不会明显变化。每次发布至少比较常见题、复杂题、值映射题和新 Schema 题，同时报告延迟与成本。

新版本先接影子流量。它接收同样的问题并生成 SQL，但不把结果交给用户。在固定快照上执行以后，再与旧版本逐题比较。只有关键切片没有回退、P95 和扫描量没有超过预算，才扩大真实流量。

Databricks Genie 把真实问题、SQL 人工审查、Benchmark 分数和线上监控放在同一个循环中。Google Cloud 的生产方案也强调用查询历史、业务上下文和评测集反复校准。[Databricks 监控与评测](https://docs.databricks.com/aws/en/genie-agents/monitor) · [Google Cloud 的评测建议](https://cloud.google.com/blog/products/databases/techniques-for-improving-text-to-sql)

本文项目可以设五个运行目标：每周新增 30～50 条分析师复核的失败样本；90% 以上线上错误能够分类；新版本关键切片回退不超过 1 个百分点；Schema 或指标变化后 24 小时内完成回归；连续四周整体正确率波动不超过 2 个百分点。

这些数字不会直接带来一次准确率跃升。它们保证三个月以后，我们仍然知道系统为什么变好，又为什么变差。

## 实现顺序

第一周先收集 200～400 个真实问题，固定数据库快照，建立结果级评测，同时跑出全量 DDL 基线。

第二到三周整理最常用主题域的语义层，完成表、字段、关系和值检索。Schema Recall 没有稳定超过 95% 时，继续修检索。

第四周接入已验证样例和结构化规划，把解析错误、执行错误和结果断言失败分开记录。

第五周只对复杂题开启多候选，计算每提高 1 个百分点需要多少 Token、延迟和数据库扫描量。

上线以后，按问题频率把口径稳定的查询沉淀为可信 SQL。等高质量修正样本超过 1,000 条，再用影子评测判断 SFT 或 RLVR 是否比当前编排更划算。

回到开头的问题，从 0 到 1 做 Text2SQL，第一步是建立一套能判断结果对错的评测。模型选择排在它后面。

之后每一次优化都只回答一个具体问题：模型有没有看到正确的表，用户值有没有映射正确，复杂问题有没有拆清楚，SQL 结果能不能被验证，高频问题是否值得继续生成。

这些问题一个个解决以后，正确率才会从 50% 走到 90%。
